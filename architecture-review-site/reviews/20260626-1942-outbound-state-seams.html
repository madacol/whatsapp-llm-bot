Problem

The queue interface is shallow. Callers and sibling modules must know whether outbound work is text or an OutboundEvent, how stream finalization works, which Baileys failures are recoverable, when to persist, how output visibility is reloaded, and how queued message handles resolve.

Solution

Move send-or-queue, replay, queue payload validation, recoverable-error policy, output-visibility loading, and queued handle resolution behind one WhatsApp Transport module. Keep `WhatsAppDeliveryPlan` as the Presentation artifact and keep Baileys execution in the delivery executor.

Benefits

  • Locality: at-least-once delivery policy lives in one place.
  • Leverage: callers submit outbound work without retry taxonomy knowledge.
  • Tests can cover the durable delivery seam from socket loss to replay.
  • Queue storage stops mirroring broad OutboundEvent validation rules.

Problem

Session state is stored in chat config fields named `harness_session_id`, `harness_session_kind`, `harness_session_history`, and `harness_fork_stack`. Those storage names are valid compatibility edges, but the internal interface also exposes them through store methods and Agent Runtime wiring.

Solution

Create a small Agent Runtime owned Session persistence module that translates legacy config storage into Session vocabulary. Agent Runtime should ask for active Session, archived Sessions, fork stack operations, and clear/restore behavior without knowing the chat-config field layout.

Benefits

  • Locality: storage migration and legacy names stay concentrated.
  • Leverage: Agent Runtime code moves toward Session and Invocation vocabulary.
  • Tests can target Session behavior instead of config field choreography.
  • The deletion test says the module earns its keep: deleting it spreads translation back to callers.

Problem

The HTTP API Transport is concrete and intentionally explicit, but `http-api-transport.js` now mixes wire concerns with turn lifecycle and event-stream state. The ledger module is useful, yet its interface is shallow because the transport still orchestrates record status, active turn lookup, wait responses, and SSE delivery.

Solution

Move the HTTP API turn flow behind a deeper transport-internal module. Keep HTTP request parsing and auth in the wire module, while turn records, event append, wait-mode response text, and stream fanout live together.

Benefits

  • Locality: event cursor and active-turn bugs are isolated.
  • Leverage: route handlers become thin wire adapters.
  • Tests can cover turn/event behavior without opening sockets.
  • Future prompt support has one place to join the flow.

Top recommendation

Start with WhatsApp outbound durability.

It is the best first improvement because it protects user-visible delivery behavior while staying inside one concrete Transport. The current module set already has real seams - delivery plans, queue storage, replay, and queued handles - but the policy is spread out enough that each caller still needs implementation knowledge. A deeper module would improve locality immediately without reopening ADR 0003 or ADR 0004.
