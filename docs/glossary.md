# Glossary

- **Transport**: An architectural role for an inbound/outbound adapter. Transports convert external messages into app-owned ChannelInputs and render app-owned outbound events; they must not carry provider-specific agent runtime details through the system. Concrete transport docs live under `docs/transports/`.
- **Channel**: The app-owned addressable surface where input arrives and output is shown. A Channel maps to whatever addressable surface a concrete Transport exposes. Channel is presentation/intake identity, not agent memory. Current code still uses `chatId` for much of this role; new app-level interfaces should prefer `channelId`.
- **ChannelInput**: One normalized inbound user input batch from a Channel. `ChatTurn` remains only as deprecated compatibility vocabulary; use ChannelInput in new docs/interfaces.
- **Input Orchestration**: The app-owned layer that handles ChannelInput intake, channel/workspace/session policy, routing, and agent invocation initiation after a transport has normalized an inbound message.
- **Media Input Enrichment**: The app-owned module that creates provider-ready text from image, audio, video, and quoted media blocks. It owns media-to-text model selection, recursive content traversal, context-reset-aware prompt context, generated image/video alt text, generated audio transcript text, and transcription status callbacks.
- **Session**: Resumable agent context/history. ACP `sessionId`, stored harness session IDs, and resume cursors are Session identity or Session references. A Session is not a Channel and not one execution. Subagents may run in their own Sessions; presentation may still show their events mixed into the parent Channel with clear subagent identity.
- **Invocation**: One agent execution inside a Session. Current code often calls invocation IDs `turnId`, and provider docs may use turn vocabulary. Treat those names as legacy or external aliases; use Invocation vocabulary in new agent-runtime docs/interfaces when referring to one execution rather than inbound user input.
- **Agent Runtime**: The app-owned module behind the Agent runtime seam that owns provider selection, runtime instance materialization, invocation lifecycle coordination, live-input injection/replay, session binding, provider invocation input, provider command execution, and provider workdir resolution.
- **Agent runtime seam**: The app-owned seam between Input Orchestration and concrete agent providers. ACP adapters live behind this seam.
- **ACP adapter**: A concrete adapter that speaks Agent Client Protocol to an agent provider such as Codex. ACP is provider/runtime protocol detail, not the app's presentation protocol.
- **Agent Run Activity layer**: The app-owned layer that converts provider/runtime activity into canonical invocation-scoped events for presentation. It owns normalization, raw provider payload interpretation, and invocation-scoped state policy such as correlation, deduplication, snapshot reconciliation, usage aggregation, and completion semantics.
- **OutboundEvent**: The canonical app-owned event model for anything intended to be presented or delivered outside the core orchestration flow. OutboundEvent covers both agent-run activity and app-originated messages. Use event names that describe the behavior; add source, cause, or policy fields only when a concrete consumer needs them.
- **Agent-run OutboundEvent**: An OutboundEvent emitted from the Agent Run Activity layer, such as reasoning, assistant output, tool activity, command activity, file changes, snapshots, usage, plans, subagents, warnings, or invocation-scoped user-input/approval lifecycle. Use this distinction only when agent-run ownership affects behavior.
- **App-originated OutboundEvent**: An OutboundEvent produced by app behavior outside an agent invocation, such as reminders, restart acknowledgements, setup replies, status replies, command errors, or plain system notifications. Use this distinction only when app ownership affects behavior.
- **Output port**: A narrow producer seam for creating OutboundEvents. Output ports are ownership-specific; they should expose semantic creation methods rather than a generic `emitOutboundEvent` escape hatch.
- **AgentRunOutputPort**: The output port given to Agent Run Activity. It creates normalized agent-run OutboundEvents and must not expose app-owned event creation.
- **AppOutputPort**: The output port given to Input Orchestration and app command modules. It creates app-owned OutboundEvents and must not expose agent-run lifecycle, tool, file-change, or usage event creation.
- **OutboundEventSink**: An infrastructure seam that accepts already-created OutboundEvents for persistence, queueing, replay, streaming, or transport delivery. It is not a domain producer seam.
- **Raw provider payload**: Diagnostic material from ACP or another provider protocol. It is input to the Agent Run Activity layer and may be retained only in provider diagnostics or raw-event logs; it must not be part of canonical OutboundEvents or Presentation inputs.
- **Diagnostic side channel**: Provider troubleshooting records kept outside canonical app events, such as raw-event logs keyed by event, invocation, item, request, or provider correlation IDs. Diagnostics may preserve raw provider payloads, but they must not define user-facing behavior. `diagnosticRaw` is adapter-to-Agent-Run-Activity input for this side channel only; it is not an OutboundEvent or Presentation field.
- **Presentation**: The layer that renders canonical app events for a concrete output surface. Presentation decides display policy, not provider payload meaning.
- **Harness**: Legacy implementation vocabulary for agent runtime modules. Treat harness naming as architecture debt to migrate in slices. Prefer Agent Runtime, ACP adapter, or Agent Run Activity vocabulary in new interfaces, docs, and touched modules.
